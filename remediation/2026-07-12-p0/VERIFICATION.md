# Проверка P0-пакета

Дата проверки: 12 июля 2026 года.

Проверка выполнялась только на локальных временных копиях. Production не
изменялся.

## Результат

- Повторная сборка после восстановления сессии побайтно совпала с сохранённым
  общим patch-файлом и манифестом.
- Общий пакет и три фазовых пакета дают побайтно одинаковые пять итоговых
  файлов.
- Каждый фазовый patch проходит `git apply --check` и применяется последовательно
  без конфликтов с исходным снимком.
- Сохранены точные резервные копии всех пяти изменяемых файлов; они побайтно
  совпадают с исходниками в `data_for_audit/site_code`.
- После применения изменяются только заявленные пять файлов.
- Четыре затронутых PHP-файла проходят `php -l` на PHP 8.1 с включёнными short
  tags.
- Полная проверка 2 046 PHP-файлов дала 2 045 успешных результатов. Единственная
  ошибка осталась в ранее известной неактивной копии
  `catalog.element/main4/result_modifier_old.php`; пакет её не затрагивает.
- Шесть сценариев canonical прошли: обычная страница, `PAGEN_1`, сортировка с
  пагинацией, одновременные `PAGEN_1`/`PAGEN_2`, первая страница и некорректное
  массивное значение параметра.
- Все 62 предложенные цели находятся в sitemap, отвечают 200 в полном обходе
  SEO-фильтров и не имеют `noindex`.
- После исправлений статический анализ `.htaccess` оставляет 15 неоднозначных
  брендовых кандидатов. Они намеренно не изменяются автоматически.
- Скрипт отказывается изменять неизменяемый снимок `data_for_audit/site_code`
  и запрещает хранить резервную копию внутри web-root.
- До первой записи скрипт сохраняет резервные копии всех файлов выбранной фазы.

## Автоматическая контрольная точка «до»

Внешний проверяющий скрипт запущен на текущем production до изменений с
ограничением 5 запросов/с. Он ожидаемо обнаружил все 64 контролируемых
расхождения:

- 0 из 62 редиректов ведут в новую одноимённую цель; все источники отвечают
  301, все предлагаемые цели — 200, причина каждого сбоя только
  `location_mismatch`;
- пагинация отвечает 200, но canonical ведёт на базовую категорию вместо
  self-canonical;
- два запроса к главной с интервалом две секунды получили разные
  `Last-Modified`, оба ровно на 60 секунд старше текущего времени ответа.

Результат сохранён локально в
`reports/generated/p0-production-baseline-before-deployment.json`. После каждой
фазы этот же скрипт должен показать соответствующее уменьшение числа ошибок, а
после полного пакета — ноль.

Успешный сценарий проверен отдельно через изолированный HTTP mock без обращения
к production: 62 из 62 редиректов, canonical и `Last-Modified` прошли,
итоговое число ошибок — 0. Таким образом проверены обе ветки самого
контрольного инструмента: текущая отрицательная и ожидаемая после исправлений.

## Что ещё нельзя считать проверенным

- Полный runtime Bitrix: в снимке нет рабочей БД, ядра и конфигурации модулей.
- Поведение композитного кеша после изменения шаблона.
- Реальная нагрузка и 5xx без access/error-логов сервера.
- Результат переобхода Google и Яндекса после изменения canonical.

Поэтому перед production обязательны staging с рабочей копией БД, очистка кеша
штатным способом, визуальная проверка основных типов страниц и отдельное
подтверждение владельца на каждую фазу.

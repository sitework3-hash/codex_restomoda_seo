# P0-пакет исправлений Restomoda

Пакет предназначен для проверки на копии сайта. Он не применялся к production.

## Что входит

1. Удаление ложного динамического `Last-Modified`.
2. Один серверный canonical; валидные `PAGEN_N` получают self-canonical, а
   сортировки и служебные параметры исключаются.
3. Удаление второго canonical из тела стандартного шаблона пагинации.
4. Удаление неработающего JavaScript, который пытался менять canonical.
5. Удаление открытого пароля из комментария заказа.
6. Исправление только тех старых брендовых 301, для которых живая проверка
   подтвердила рабочую одноимённую цель с HTTP 200.

Автоматически не исправляются:

- редиректы, у которых одноимённая цель не отвечает 200;
- Amitek `/info/brands/amitek/?section_id=1077`, потому что готовой брендовой
  посадочной страницы нет;
- 5xx и недоступность `robots.txt` без серверных логов;
- генерация sitemap без настроек модулей Bitrix.

## Сборка без изменений файлов

Запускать локально из корня SEO-проекта:

```bash
.venv/bin/python scripts/build_p0_remediation.py
```

Команда работает в dry-run и создаёт только:

- `reports/generated/p0-remediation.patch`;
- `reports/generated/p0-remediation-manifest.json`.

Пакет также разделён на три независимые фазы:

- `p0a-safety.patch` — пароль и `Last-Modified`;
- `p0b-redirects.patch` — 62 подтверждённых брендовых 301;
- `p0c-canonical.patch` — изменение политики пагинации и canonical.

Такой порядок уменьшает риск: сначала выкладываются локальные безопасные
изменения, затем редиректы, а canonical — отдельно после проверки staging.

Пересборка отдельной фазы:

```bash
.venv/bin/python scripts/build_p0_remediation.py --scope safety
.venv/bin/python scripts/build_p0_remediation.py --scope redirects
.venv/bin/python scripts/build_p0_remediation.py --scope canonical
```

Только фазы `redirects` и `all` требуют сохранённый результат живой проверки
`reports/generated/htaccess_redirect_analysis.json`.

## Проверка на временной копии

Запускать локально из корня SEO-проекта:

```bash
cp -a data_for_audit/site_code /tmp/restomoda-p0-site
.venv/bin/python scripts/build_p0_remediation.py \
  --site-root /tmp/restomoda-p0-site \
  --write \
  --backup-dir /tmp/restomoda-p0-backup
```

После применения необходимо проверить синтаксис изменённых PHP-файлов и
сравнить их с резервной копией. Production-выкладка выполняется только после
отдельного подтверждения владельца, резервной копии и теста на staging.

Скрипт технически запрещает `--write` в `data_for_audit/site_code`: исходный
снимок аудита должен оставаться неизменным. Папка резервной копии также обязана
находиться вне корня сайта. Для каждой фазы используется отдельная пустая папка
резервной копии.

## Условия отката

Скрипт при `--write` требует пустую папку `--backup-dir` и сохраняет туда
оригиналы всех пяти изменяемых файлов. Откат выполняется восстановлением
этих файлов с сохранением владельца и прав, затем очисткой кеша Bitrix штатным
способом. Production-команды будут подготовлены отдельно после уточнения
пользователя сервера, корня сайта и способа выкладки.

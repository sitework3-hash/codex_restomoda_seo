# Порядок тестовой и production-выкладки

Production пока не изменялся. Ниже — обязательный порядок, а не разрешение на
выкладку.

## Что нужно уточнить до начала

- SSH-пользователь для выкладки (часто `bitrix`, но это нельзя предполагать).
- Точный корень сайта; по коду ожидается `/home/bitrix/www`, его нужно
  подтвердить на сервере.
- Есть ли staging с рабочей копией БД и ядра Bitrix.
- Как штатно очищается кеш: админ-панель, CLI или deployment-скрипт.
- Где хранить резервную копию вне web-root.
- Где находятся nginx/Apache/PHP-FPM access/error-логи.

Перед каждой фазой нужен отдельный Git-коммит или архив пяти исходных файлов,
фиксация их владельца/группы/прав и проверка свободного места.

## Фаза P0a — безопасность и HTTP freshness

Файл: `p0a-safety.patch`.

Изменяет два файла:

- `local/templates/aspro_next/header.php`;
- `local/php_interface/init.php`.

Проверки на staging:

1. `php -d short_open_tag=1 -l` для обоих файлов.
2. Два последовательных запроса к главной, категории и товару: динамический
   `Last-Modified` больше не должен меняться каждую минуту.
3. Создание тестового заказа через форму 14: комментарий не должен содержать
   пароль.
4. Проверка входа/штатного восстановления пароля тестового пользователя.

Фаза откатывается при PHP warning/fatal, ошибке формы, 5xx или проблемах входа.

## Фаза P0b — подтверждённые 301

Файл: `p0b-redirects.patch`.

Изменяет только `.htaccess`: 62 источника должны дать один 301 на одноимённую
цель, затем конечный 200. Пятнадцать неоднозначных правил остаются без
изменений.

Проверки на staging:

1. Проверить синтаксис конфигурации штатной командой используемого веб-сервера.
2. Проверить главную, каталог, товар, robots и sitemap на отсутствие общего
   цикла/массового редиректа.
3. Прогнать все пары из `p0b-redirects-manifest.json`.
4. Убедиться, что нет цепочек, целей 404 и переходов на другой бренд.

Фаза откатывается немедленно при 5xx, цикле, массовом редиректе или несовпадении
любой из 62 целей.

Автоматическая внешняя проверка после staging/production:

```bash
.venv/bin/python scripts/verify_p0_deployment.py \
  --base-url https://staging.example/ \
  --pagination-url 'https://staging.example/catalog/teplovoe-oborudovanie/?PAGEN_1=3'
```

## Фаза P0c — canonical пагинации

Файл: `p0c-canonical.patch`.

Изменяет три файла шаблона. Это самая значимая SEO-фаза, поэтому она
выкладывается отдельно после P0a/P0b и проверки staging.

Нужно проверить исходный HTML, а не только DOM браузера:

- главная, категория, товар и SEO-фильтр имеют ровно один canonical в `<head>`;
- `?PAGEN_1=3` имеет self-canonical с `PAGEN_1=3`;
- `?sort=PRICE&PAGEN_1=3` не переносит `sort` в canonical;
- `PAGEN_1=1` редиректится на базовую страницу либо получает canonical без
  параметра;
- в `<body>` нет второго canonical;
- robots/noindex SEO-фильтров не изменились.

После production нужно зафиксировать дату выкладки и следить минимум 4–6
недель за `NOT_CANONICAL`, числом страниц в поиске, показами и логами роботов.

## Production

Для production требуется отдельное явное подтверждение владельца. Рекомендуемый
порядок: P0a, наблюдение не менее суток; затем P0b, наблюдение; затем P0c.
Выкладку лучше выполнять в период низкой нагрузки, но без включения режима
обслуживания, если штатный процесс позволяет атомарно заменить файлы.

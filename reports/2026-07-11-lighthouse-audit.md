# Lighthouse-аудит производительности Restomoda

Дата проверки: 11 июля 2026 года.

Инструмент: локальный Lighthouse 13.4.0 и установленный Google Chrome 149.
Проверка лабораторная: она моделирует загрузку страницы и не заменяет полевые
Core Web Vitals реальных посетителей из Google Search Console.

## Резюме

Мобильная производительность критически низкая на всех проверенных шаблонах.
Основное время теряется не только на ответе сервера, но прежде всего на
JavaScript, обработке главного потока и отрисовке. Карточка товара дополнительно
загружает около 6,3 МБ ресурсов и делает около 200 сетевых запросов.

## P0. Критический LCP на мобильных шаблонах

На всех мобильных страницах LCP находится в диапазоне **9,2–13,6 секунды**.
Целевое значение Core Web Vitals — не более 2,5 секунды для большинства
реальных посещений.

Результаты показывают, что проблема общая для сайта, а не ограничена одной
категорией. Главная на desktop заметно быстрее, что дополнительно указывает на
перегрузку мобильного сценария при эмуляции более слабого устройства.

Lighthouse оценивает потенциальное сокращение неиспользуемого JavaScript:

- категория — около 326 КБ;
- SEO-фильтр — около 288 КБ;
- товар — около 585 КБ.

### Что исправить

1. Загружать JavaScript по типу страницы, а не единым набором для всего сайта.
2. Отложить виджеты, отзывы, форумные компоненты, карты, аналитику и формы до
   взаимодействия или появления блока в области просмотра.
3. Разделить крупные bundles Aspro/Bitrix и удалить неиспользуемые компоненты.
4. Профилировать долгие задачи Chrome, особенно инициализацию каталожного
   фильтра и карточки товара.
5. После исправления повторить Lighthouse на тех же URL и тех же версиях Chrome.

## P0. Карточка товара загружает около 6,3 МБ

Карточка Hicold TS 16 GN выполнила около **200 запросов** и передала примерно:

- изображения — 3,75 МБ;
- JavaScript — 1,75 МБ;
- CSS — 451 КБ;
- HTML-документы — 180 КБ переданных данных;
- общий вес — около 6,3 МБ.

Lighthouse оценивает около 585 КБ неиспользуемого JavaScript и 401 КБ
неиспользуемого CSS. Дополнительно найдены JavaScript-ошибки в консоли и
неправильные пропорции отдельных изображений.

### Что исправить

1. Не загружать полноразмерные изображения галереи до открытия.
2. Использовать корректные `srcset`, `sizes`, размеры контейнера и современные
   форматы.
3. Загружать блоки отзывов/форума и их скрипты по требованию.
4. Устранить 17 запросов 404 к `bx_smile_*.png/` и ошибку `addExt`, найденные
   предыдущей Chrome-проверкой.
5. Проверить, почему Lighthouse видит четыре запроса типа Document на одной
   карточке.

## P1. Неиспользуемый CSS

Оценка потенциального сокращения:

- главная — около 224 КБ;
- категория — около 227 КБ;
- SEO-фильтр — около 224 КБ;
- товар — около 401 КБ.

Необходимо выделить критические стили первого экрана и разделить CSS по
шаблонам. Удалять правила автоматически без визуальной регрессии нельзя.

## P1. Категория имеет плохой CLS

У категории тестораскаточных машин лабораторный CLS равен **0,253** — это выше
порогового значения 0,1 и находится в плохой зоне.

Следует зарезервировать размеры изображений, баннеров, фильтра, cookie-окна и
динамически вставляемых блоков до их загрузки.

## P1. Некраулируемые ссылки

Категория и SEO-фильтр получили SEO 92 из-за ссылок вида:

```html
<a class="bx_ui_slider_handle" href="javascript:void(0)">
```

На карточке товара дополнительно найдены `<a>` без нормального `href` для
прикрепления файла и быстрого заказа.

Элементы управления, которые не являются переходами, должны быть `<button>`.
Настоящие ссылки должны иметь обычный индексируемый `href`. Это улучшит
семантику, доступность и оценку SEO.

## P2. Доступность и Best Practices

Повторяются следующие проблемы:

- интерактивные элементы внутри `aria-hidden`;
- недостаточный цветовой контраст;
- iframe без `title`;
- слишком маленькие или близко расположенные сенсорные цели;
- отсутствие основного landmark `<main>`;
- запрет вставки в поля;
- сторонние cookies;
- устаревшие браузерные API;
- отсутствующие source maps для крупных собственных скриптов.

Карточка товара также содержит поля без связанных `<label>` и ошибки консоли.

## Серверный ответ

В Lighthouse корневой документ отвечал примерно за 400–900 мс, но полный обход
15 880 SEO-фильтров показал медиану 1,95 с и 95-й процентиль 3,33 с. Разница
объясняется кешем и разными категориями, поэтому оптимизация нужна на обоих
уровнях:

1. серверный кеш и запросы Bitrix;
2. клиентский JavaScript, CSS и ресурсы.

## Критерии приёмки

После исправлений на тех же URL:

- мобильный LCP ≤ 2,5 с;
- лабораторный TBT ≤ 200 мс;
- CLS ≤ 0,1;
- отсутствуют ошибки консоли и запросы 404;
- нет `<a href="javascript:void(0)">`;
- карточка товара заметно сокращена по весу и числу запросов;
- полевые данные Search Console показывают «хорошо» для LCP, CLS и INP.

## Ограничения

- это единичные лабораторные запуски, показатели могут колебаться;
- PageSpeed API без отдельного ключа вернул 429 из-за нулевой квоты среды;
- полевые CrUX/Core Web Vitals необходимо получить из Google Search Console;
- тест не менял сайт и не проверял сценарий авторизованного пользователя.

Машинные JSON-результаты находятся локально в
`reports/generated/lighthouse/` и исключены из Git.
